Collection of Lot Check Rejection Examples (2007/12/25 Public Version)

We have collected a list of issues that frequently result in rejection during Lot Check (Nintendo's master guidelines conformity check) for Nintendo DS. To avoid possible rejection during Lot Check, please review this list before submitting your game.

This list of issues can also be viewed at NTSC-ONLINE. (Nintendo DS Support Page -> Technical Information -> Collection of Lot Check Rejection Examples)

• Game Card Interrupts During Sleep Mode

Problem:
"Game Card interrupt generation or Game Card removal" is specified as one of the conditions for transition from Sleep Mode to Active Mode. Therefore, if a Game Card is removed by mistake, or if a Game Card that generates a Game Card interrupt is inserted (for example, a Game Card that uses advance movies or RTC), then the system will move from Sleep Mode to Active Mode.

Solution:
Do not include "Game Card interrupt generation or Game Card removal" as a condition for transition from Sleep Mode to Active Mode. Specifically, do not specify PM_TRIGGER_CARTRIDGE as a recovery trigger for the PM_GoSleepMode function.

Reference:
DS Programming Guidelines (Version 2.0.5) 5.2.2 Sleep Mode to Active Mode Transitions [Required], NITRO-SDK Function Reference "PM GoSleepMode"

(Added 2006/03/24)

• DS Does Not Always Enter Power Conservation State when Closed

Problem:
The DS does not enter power conservation state even if the DS is closed during DS Wireless Communications or Wi-Fi communications.

Solution:
If the DS is closed, be sure to move the DS into one of the three power conservation states (Sleep Mode, LCD OFF state, or backlight OFF state). Unless it is difficult to implement, move to a state with the best power conservation benefits. However, entering Sleep Mode is prohibited when the DS is closed during Wi-Fi communications. In this case, move to either the LCD OFF state or the backlight OFF state.

Reference:
DS Programming Guidelines (Version 2.0.5)
5.6.1 Power Conservation when the DS is Closed [Required], Nintendo Wi-Fi Connection Programming Guidelines for DS (Ver.1.3.4)
7.3 Prohibiting Transition to Sleep Mode During Communications [Required]

(Updated 2007/12/10) Changed item names and contents. (Added 2005/09/07)

• DS Enters Sleep Mode While Writing to Backup Memory

Problem:
If the DS is closed while writing to backup memory, it will enter Sleep Mode without waiting for the writing of data to complete. In particular, if several write operations are performed internally during a single save, the DS may enter Sleep Mode when the DS is closed between these write operations.

Solution:
When deleting or writing data to or from backup memory, be sure that the DS waits for processing to complete and then enters Sleep Mode.

Reference:
DS Programming Guidelines (Version 2.0.5), 5.2.3 Mode Transitions During Backup and Communication [Required]

(Added 2007/12/10)

• Processing when Backup Memory Read Errors Occur

Processing when accessing backup memory and the Read function returns an error

Problem 1:
The error message indicates that backup data is corrupted (not present), even though there is no problem with the actual backup data.

Problem 2:
The error message is missing the expression "Turn off power," consequently prompting the user to hot swap.

Problem 3:
Although the correct error message is displayed, the game does not stop.

Solution:
The Read function can sometimes return an error because of a poor connection with the card terminals and other causes. In this case, since there is a possibility that the problem can be improved by reinserting the DS Card, display a message such as: "The save data could not be accessed. Please turn the power off and reinsert the DS Card." Then have the game enter a Halt state. Do not mistakenly determine that backup data has been corrupted. Also, take care that the message does not prompt the user to hot swap. It is critical to confirm the backup memory and provide processing for read errors, especially when the power is being turned on.

Verification:
On the IS-NITRO-EMULATOR, while a program is executing, with the power on for the DS Card Slot but with no DS Card in the slot, you can intentionally generate a poor connection between the DS Card connector and the DS Card.

Reference:
DS Programming Guidelines (Version 2.0.5), 2.5.13 Caution About Reading from Backup Memory [Required], 2.5.14 Verification of the DS Card Backup Memory [Required]

(Updated 2007/12/10) Added examples of problems. (Added 2005/09/07)

• Operations when Backup Data is Corrupted

Problem:
Malfunction occurs when corrupted backup data is read.

Solution:
Backup data can be corrupted by turning off the power during initialization or writing of backup data.
Therefore, even when backup data values are corrupted, the program should handle them without malfunctioning. Use a checksum or other method to check the integrity of backup data, and if the data is corrupted and cannot be restored, inform the user of this. Then, if it cannot be restored, provide the option to delete it.

Verification:
Using the IS-NITRO-EMULATOR, read backup memory data from the card and destroy the active area using a general binary editor or by other means. Write the corrupted backup data to backup memory and start the card. The mb_card_backup_test tool provided by Nintendo allows you to randomly corrupt backup memory. However, use it for a simplified check.

Reference:
DS Programming Guidelines (Version 2.0.5), 2.5.6 Backup Data Reliability [Required]

(Updated 2007/12/10) Modified the Verification. (Added 2005/09/07)

• An Error Message is Displayed for Backup Data when Shipped from the Factory

Problem:
An error message such as "Backup data corrupted" is displayed when the DS is started using the factory-shipped backup data.

Solution:
Include a mechanism to determine whether backup memory data is in the factory default state or the data corrupted state. For example, you can record a fixed value (magic number) in a specific location of backup memory, then you can display an error message only if the value is maintained and if the backup data is corrupted.

Reference:
DS Programming Guidelines (Version 2.0.5), 2.5.3.2 Disabling the Display of Error Messages for Default Backup Data

(Added 2007/12/10)

• Display While Writing to Backup Memory

Problem:
There is no display indicating that backup memory is being written, even though the writing time to backup memory exceeds 0.5 seconds.

Solution:
If the writing time to backup memory exceeds 0.5 seconds, be sure to make a display that indicates that backup memory is being written. At this time, take care that the display does not disappear before writing is complete. However, it is okay for the display time to be longer than the actual write time to improve readability. Also, if the writing time to backup memory exceeds 5 seconds, be sure to animate some part of the screen so that the player does not mistake this for a hang-up.

Reference:
DS Programming Guidelines (Version 2.0.5), 2.5.10 Display While Writing to the Backup Memory [Required]

(Added 2005/12/10)

•GGID

Problem:
Despite incompatibility between titles localized for overseas markets through DS wireless communications, the same GGID is used.

Solution:
When DS wireless communications are not intended between the same title in different markets, use the GGID issued for the specific market.

(Added 2005/07/07)

•Updating TGID

Problem 1:
Each time a system becomes a DS Download Play parent during a game, the TGID is not updated and remains a fixed value.

0001 > 0001 > 0001 > Turn off the power and restart > 0001 (The same TGID value)

">" represents terminating communication during game play and resuming communication as a parent without turning off power.

Problem 2:
Each time a system becomes a DS Download Play parent during a game, the TGID is updated, but if the power is turned off and the system becomes a DS Download Play parent again, the TGID is the same value.

0001 > 0002 > 0003 > Turn off the power and restart > 0001 (The same as the first value)

Solution:

In NITRO-SDK 2.1 and later, when using the MB_Init function, specify the constant MB_TGID_AUTO for the tgid argument, and the TGID value will be updated each time the parent power is turned on.
In versions older than NITRO-SDK 2.1:
Set the TGID using the OS_GetVBlankCount function or RTC time data, so that the TGID value will be updated each time the parent power is turned on.


Reference:
NITRO-SDK Function Reference (Multiboot Library (MB)) DS Programming Guidelines (Version 2.0.5) 6.3.8 TGID Used in Single-Card Play [Required]

(Added 2005/07/07)

• Graphics for Icons Used for Communications

Problems:

• An enlarged DS Wireless Icon is displayed.
• Because the background is dark, the colors on the DS Wireless Icon are changed.
• In DS Download Play, the Reception Strength Icons are not displayed on the parent, even though it is sending data to a child.
• Some of the three Reception Strength Icon colors are changed.
• The Reception Strength Icons, which should have three levels (green > yellow > red), only have two levels (green > red).
• Although linked, the Reception Strength Icons are hidden behind a popup display or other obstruction for a long period.

Solution:
Currently three types of Nintendo-specified icons are being distributed: the DS Wireless Icon, Reception Strength Icons, and Chat Icon. (Icon image data is located in the $(NITROSDK_ROOT)/data/ directory of the NITRO-SDK package.)

Aside from adjusting slightly the tint of green, yellow and red colors of the Reception Strength Icons, it is not permissible to make changes to these icons, so problems like those above will have to be fixed.

Reference:
DS Programming Guidelines (Version 2.0.5) 6.1.4 DS Wireless Communications ON State [Required], 6.2.1 Reception Strength Icon [Required], 6.4.3 Chat Icon [Required]

(Added on 2007/01/31) Added the location of icon image data.
(Added 2005/07/07)

• Using a Non-Distributed GGID

Problem 1:
The GGID used for DS Download Play is a private GGID (0x003FFF00 to 0x003FFFFF).

Solution:
Make sure you are using the GGID that was assigned by Nintendo.

This GGID value can be found in the comments section of the Development Plan document.

Reference:
DS Programming Guidelines (Version 2.0.5) 6.3.7 GGID Application [Required]

(Added 2005/09/07)

• DS Wireless Communications Turn on Without Confirming with the Player

Problem:
After communications are canceled by the player, or DS wireless communications are automatically turned off due to a communication error, DS wireless communications enter a sequence that turns communications on without confirming with the player.
Example of Sequence Resulting in a Problem:
(1) A screen that includes a "DS wireless icon" or "message-based confirmation" when turning on DS wireless communications,
(2) A screen for selecting parent or child and setting game rules (here, actual wireless communications are ON),
(3) A screen when searching for communication partners
Note: If a communications error occurs due to canceled communications or other reason after this screen, DS wireless communications is turned off and the display returns to the previous screen, (2).

Solution:
When wireless communications are turned off due to communication cancellation by the player or a communications error, be sure to return to a state before the confirmation with the player (using a message or icon). By doing this, it is possible to carry out confirmation with the player by passing through a screen that includes a DS wireless icon (or message-based confirmation), even if the player turns DS wireless communications ON again.

Reference:
DS Programming Guidelines (Version 2.0.5), 6.1.4 DS Wireless Communications ON State [Required]

(Added 2007/12/10)

• Processing when Card Removal is Detected

The following applies to titles already released that were made with old versions of the NITRO-SDK. Be especially careful if you are making ROM revisions or localized products.

Problem 1:
When the DS Card is removed during Sleep Mode, the power remains on.

Solution:
When using library versions NITRO-SDK 2.0 RC3 and later, the power can automatically be turned off by including DS Card interrupts in the conditions for recovering from Sleep Mode.
If the above SDK version or later cannot be used, then after recovering from a DS Card interrupt, perform power shutdown processing.

Problem 2:
If a DS Card is removed during DS Wireless Communications, DS Wireless Communications does not stop.

Solution:
If you are using the NITRO-SDK 2.0 library version RC4 or later, this processing is automatic. If the above SDK version or later cannot be used, when the DS Card is removed, use the WM_End function to shut down DS Wireless Communications.

Hot-swapping is forbidden for Nintendo DS, even though the hardware is capable of hot-swapping.
In other cases, if you detect that a DS Card or Game Pak was removed during a game, handle it in a way appropriate to the particular situation.

Reference:
DS Programming Guidelines (Version 2.0.5) 2.2.1 Processing when DS Card Removal is Detected [Required]

(Added 2005/07/07)
